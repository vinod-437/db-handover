Salary Slip Process Flow

Technical analysis of Database Functions: uspgeneratesalaryslip & uspgetreportfields

Process Flow Architecture

The following diagram maps out how an employee's salary slip is generated from input arguments down to the multiple refcursors returned to the application backend.

Main uspgeneratesalaryslip Detailed Logics

  • Inputs: Takes p_emp_code, p_month, p_year, and origin flags.
  • Initialization: Gathers foundational data like limits (MTD), Payout Mode, Principal Employer name, and the customer IDs.
  • Edge Cases validation: Forces exceptions for "Special categories", "Hidden Salary slips" enabled by app settings, and threshold limits based on clients (e.g. CTC <= 22000 for Jivo or CTC <= 8000 for Kateba).
  • Fetching Financials: It aggregates month-wise data by wrapping uspmonthwisebilling. tmp_monsalslip gathers totals, leaves, PF, ESI, and deductions.
  • Complex Multi-cursor returns: Uses PostgreSQL's SETOF refcursor feature returning around 8 distinct cursors respectively for: Main Slip Details, Deductions breakdown, Earnings Breakdown, PAN, Header Templates, and Defaults.

Sub uspgetreportfields Detailed Logics

  • Dynamic Generation: Handles the dynamic loading of labels to ensure the UI headers reflect the correct database columns.
  • Integration: Depends on getmastersalarystructure to fetch base earning/deduction rules attached to the customer account id.
  • Rate Cleansing & Overrides: Loops through generated components, dynamically appending textual flags like 'Arrear ' and 'Rate '. If the report is expressly a 'Salary Slip', it purges rate-based columns.
  • Client Overrides: Hardcoded custom formatting rules. For instance, customer 7653 specifically updates "ESIC" labels to "ESI Employee Contribution @.75%".
  • Output format: Returns sorted refcursor reportcolumnname, reportcomponentname, displayorder to be utilized by the parent.

Sub uspmonthwisebilling Detailed Logics

  • Action Routing: Depending on p_action, it either generates month billing (GetMonthBilling) or arrear billing by routing to uspmonthwiseiablityreport.
  • Complex Aggregation: Harvests base metrics from tbl_monthlysalary and cross-references missing or updated deductions such as Conveyance, Tea Allowance, and Advances.
  • Arrears Processing: Consolidates data from previous/current wages explicitly targeted for increments and arrears.
  • Outputs: Uses a monolithic temporary table tmpbilling to structure over 100 column properties (Earnings, Deductions, Transfers, Ledgers) dynamically feeding into the main salary generation cursor.
