This procedure is designed to dynamically accept a custom JSON payload defining an employee's salary breakdown, and securely translate it into a compliant, mathematically balanced gross/net salary structure.

Phase 1: Setup, Safety, & Validation

  • Apprentice Validations: If the employee's Job Type is 'Apprentice' or 'Trainee', the system acts aggressively: it forces all compliance (PF, ESI, PT, LWF, Gratuity) to be disabled. If any are opted in, it outright blocks execution and throws an error.
  • Lock-in Rules: System queries historical generation tables. If PF or ESI was already generated for them in the past (or UAN/ESI numbers generated), they are blocked from disabling them now.
  • Deductions Cache: Grabs required upfront deduction values (Meal Vouchers, Fixed and Variable quarterly/annual deductions, Taxable & Non-Taxable bonuses).

Phase 2: Data Extraction & Accumulation

  • JSON Unpacking: The provided p_salarystructure parameter (which is a JSON string) is extracted directly into a temporary SQL table (tmpsalarycomponent) so SQL math functions can interact with the dynamic components natively.
  • Master Comparisons: Loops against the core Employer setups to decide exactly which of those custom fields should attract Gratuity.
  • Summation Matrix: Maps down column by column to extract explicit values for Basic, HRA, Target Allowances, Conveyance, Education Allowances, Overtime, Leave Encashment, etc.

Phase 3: Core Compliance Calculation

  • Minimum Wage Blocker: For companies > 20 Employees who disable PF/ESI, the procedure enforces hard stops on Basic Pay (≤ ₹15k) and CTC (< 17.4k).
  • State Demands: Labour Welfare Fund (LWF) matching logic extracts employer and employee obligations divided by their frequency (monthly, annually, etc.).
  • ESI Logic: Identifies all components marked as "Taxable". If their sum is ≤ ₹21,000, or the employee inherently triggers ESI, it calculates the ESI Employer (3.25%) and Ee (0.75%) figures.
  • Provident Fund (EPF): Evaluates strictly selected "PF Applicable components" to safely calculate the 12% deduction, watching for EDLI overhead caps.

Phase 4: Synthesis & Output Compilation

  • Initial Wrap: Mathematical summations compute straightforward Target CTC (Gross + Liabilities + Overheads).
  • Business Overrides (The V2.7 Fix): Standard systems assume Gross is simply [Basic + HRA + Allowances]. Here, it checks if the Business Setup explicitly dictates which custom JSON components are counted as Gross (using includedingross flag).
  • Net Tuning & Charity: Calculates Net Take-Home Pay. From there it explicitly scrubs Charitable contributions, Voluntary PF (VPF), and non-taxable bonuses.
  • Serialization: Compiles all computed individual allowance objects back into a clean nested JSON object (`v_salarymasterjson`).
  • Conclusion: Spits out the massive refcursor structure containing all ~65 variables necessary to commit the setup to the database.
