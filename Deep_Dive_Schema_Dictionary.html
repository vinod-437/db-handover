Core Logical Entity Architecture

TankhaPay uses Logical Constraints driven by application logic to prevent bulk bottlenecking, rather than strict DB-level cascading. The entire system is anchored to the Employee via emp_code and Employer via customeraccountid.

01

HRMS & Master Data

Static configuration for employees and corporate funding entities.

openappointments

  • emp_code PK
    VARCHAR / BIGINT
    The absolute central identifier. Nearly every downstream payroll calculation anchors back to this value.
  • customeraccountid FK
    BIGINT
    Binds the employee to a parent employer wallet.
    References employer.account_id
  • appointment_status_id INT
    Governs exactly where an employee is within the onboarding lifecycle. Fully verified setups are marked as '11'.
    11 = Converted (Active) 1 = Incomplete Onboarding 13 = Cancelled 15 = Suspended
  • isactive STATE FLAG
    BIT
    Logical deletion constraint preventing execution. 1 = Active, 0 = Inactive.

employer

  • account_id / employerid PK
    BIGINT
    The ultimate parent entity bounding liabilities and cash logic.
  • ach_transaction_type VARCHAR
    Global remitter configuration for banking logic routing.
  • active STATE FLAG
    BIT
    Halts all calculations universally for this corporate entity if set to 0.

empsalaryregister

  • appointment_id FK
    INT
    Binds the salary definition.
    References openappointments.emp_id
  • gross / ctc / basic NUMERIC
    Contractual numeric margins holding the final max bounds scaled by Loss-of-Pay multipliers.
  • isactive STATE FLAG
    BIT
    Used alongside 'FINYEAR' definitions to keep historical structures disabled.

mst_otherduction

Custom Setup Bounds: This sits alongside the Salary Register to map custom, non-statutory bounds (e.g., custom fines or contributions) uniquely applied per Employer.
  • customeraccountid FK
    BIGINT
    Binds this custom rule strictly to a specific Employer.
    References employer.account_id
  • masterledgername VARCHAR
    Dictates the financial bucket array this deduction routes into.
  • calc_type VARCHAR
    Governs the mathematical operation.
    Percentage Flat
  • tpactivestatus / active STATE FLAG
    VARCHAR

03

Core Payroll & Billing Engine

The Nexus where execution happens defining final mathematical balances.

tbl_monthlysalary

Architectural Fact: The final payroll block natively combines tracking for physical payouts alongside internal accounting bounds! By reading issalaryorliability, the system parses the ledger row identity.
  • emp_code FK
    BIGINT
    References openappointments.emp_code
  • recordscreen VARCHAR
    The Classifier: The literal router that categorizes what the actual computational row encompasses (Arear diff vs physical check).
    Current Wages Previous Wages Increment Arear
  • arearprocessmonth INT
    The internal temporal boundary specifying exactly which calendar month (1-12) the arrear generation is isolated against.
  • disbursedledgerids CSV-FK
    TEXT (CSV)
    The critical array binding external off-cycle amounts (Increments/Arrears) securely into standard payroll.
    References tbl_employeeledger.id
  • batchid VARCHAR
    Execution group routing ID. Forces atomic bounds around massive cohorts for easy mass deletion or commit.
  • issalaryorliability VARCHAR
    S = Salary Equivalent L = Tax/Compliance Liability
  • issalarydownloaded VARCHAR
    P = Processing Finalized/Paid H = Held NULL = Pending Disbursement
  • is_rejected VETO FLAG
    BIT
    Administrator Veto override, killing physical finalisation before the banking stage is breached. (0 = Active, 1 = Bounced).

tbl_employeeledger

The Increments & Arrears Bridge: A staging queue managing non-standard amounts awaiting physical pull into standard salaries via disbursedledgerids.
  • id PK
    BIGINT
  • emp_code FK
    BIGINT
    References openappointments.emp_code
  • masterhead VARCHAR
    Additional income Reimbursement Advance Deduction
  • isledgerdisbursed STATE FLAG
    BIT
    Locks the ledger completely out of processing the second it sweeps into tbl_monthlysalary.

managetemppausedsalary

  • pausedstatus VARCHAR
    Administrative override actively stopping the employee from passing to the banking engine.
    Enable = Actively Blocked Disable = Resumed

05

Global Master Configurations

The strictly enforced foundation matrices that orchestrate routing and boundaries.

mst_appointment_status

Supplies the master enumerated identity constraints injected directly into openappointments.

11 = Converted 1 = Onboarding 13 = Cancelled

mst_minimumwagescategory

Stores the hard compliance boundaries enforced by law that the empsalaryregister evaluates against during custom setups.

mst_taxslabs & rebates

The automated percentage and flat-rate engine strictly determining the mathematical overhead held on the employer's wallet side for taxes.

mst_customeraccount

The actual universal registry defining the primary corporate identity utilized heavily as the customeraccountid FK.

mst_paymenttype

Holds the final routing mechanisms to determine exactly which bank formatting (or cash distribution logic) takes precedence.
