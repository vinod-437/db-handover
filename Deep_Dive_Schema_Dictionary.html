Empirical Database Schema Analysis

A definitive architecture visualizer constructed directly from PostgreSQL empirical row-counts and foreign-key cluster density identifying the actual core of TankhaPay.

01

HRMS & Master Data

Static configuration for employees and corporate funding entities.

openappointments

  • emp_code VARCHAR
    Unique identifier across the system. (Top tier foreign key driver).
  • customeraccountid BIGINT
    Binds the employee to a parent employer wallet.
  • appointment_status_id INT
    Governs if an employee is active/payable. Defaults to 11.

employer

  • account_id BIGINT
    Links physical profiles to internal virtual wallets in tbl_account.
  • ach_transaction_type VARCHAR
    Global remitter configuration for HDFC/HSBC logic.
  • bank_ac_no VARCHAR
    Registered master bank account used in payout headers.

empsalaryregister

  • gross / ctc / basic / hra NUMERIC
    Contractual numeric margins bounding how loss-of-pay scripts evaluate final totals.
  • employeeepfrate NUMERIC
    Hardcoded overrides or evaluated deduction bounds.
02

Unprocessed Inputs & Staging

Timesheets and Bio-device aggregations moving towards calculations.

tbl_monthly_attendance

  • attendance_date DATE
    Granular daily logs tracing raw inputs before any calculation script touches them.
  • status VARCHAR
    Present, Absent, Leave states recorded per day.

cmsdownloadedwages

  • billable_days NUMERIC
    Calculated metric pushing directly into Pro-rata core algorithms in uspmonthwisebilling.
  • issalaryunlocked BIT
    Validation barrier flag showing if data passed HR approval flow.
03

Core Payroll & Billing Engine

The Nexus where execution happens defining final mathematical balances.

tbl_monthlysalary

Key Insight: TankhaPay does not cleanly separate "Employer Billing" from "Employee Salary". This massive table acts as a polyfill carrying real cash payouts (S) and tax funding liability records (L) collectively.
  • mprmonth / mpryear INT
    Points to the specific matrix execution block locking these ledger outputs.
  • batchid VARCHAR
    Group execution tag allowing mass commits or rollbacks on bulk reports.
  • netpay NUMERIC
    Cash-in-hand determination mapped directly to bank files.
  • issalarydownloaded VARCHAR
    Stoppers double transfers (P = Paid, N = Pending).
  • issalaryorliability VARCHAR
    Determines purpose ('S' = Transfer to employee, 'L' = Internal retention).

tbl_receivables

  • netamount NUMERIC
    The physical wallet limit for a customeraccount. Evaluated constantly.
  • entrytype VARCHAR
    Defines if the ledger row acts as an invoice debit or receipt credit.
04

Banking Pipelines

Output arrays mapping the ledger data into actual bank transfer logic.

banktransfers

  • bankbatchcode VARCHAR
    Constructs the actual HDFC/HSBC CSV text mapping tags.
  • referencenumber VARCHAR
    The absolute generated trace number appended to the bank batch file to catch rejections.
  • salary NUMERIC
    Should perfectly echo the netpay calculation stored earlier.
  • salmonth / salyear INT
    Binds the output vector backward to the specific execution block.
